# 기술 결정 이력 (ADR)

새 결정은 번호를 늘려 아래에 추가. 폐기된 결정은 상태를 "폐기"로 바꾸고 후속 ADR 번호를 적음.

---

## ADR-001: V/L/A를 머신 단위로 분리하지 않는다

**상태**: 채택

**맥락**: Mac=V, Cloud=L, Windows=A 식의 깔끔한 3분할을 검토했으나 leaky함.

**결정**:
- Mac이 V뿐 아니라 grounding/orchestration까지 책임진다
- Windows는 V·L에 관여하지 않는 pure actuator
- Cloud는 추론(L)만, 좌표는 보지 않음

**이유**:
1. 좌표 그라운딩은 비전 결과를 들고 있는 쪽에 살아야 함 → Mac
2. L이 V를 도구처럼 부르는 패턴(hover-후-재관찰)에서 V↔L 왕복이 Mac 내부에 머물러야 비용·지연 폭발 방지
3. 진짜 VLA의 'A'는 모터 토큰 레벨. PyAutoGUI는 그 뒤의 드라이버.

---

## ADR-002: Reflection은 AI 자율 결정 + 시스템 5개 안전망

**상태**: 채택

**결정**: AI가 상태 표현 양식·갱신 시점·기록 내용을 모두 자율 결정. 단 시스템은 다음 5개를 강제한다.

1. **스키마 표류 인지**: 상태 파일 덮어쓸 때 *이전 키 ↔ 새 키 diff*를 다음 프롬프트에 끼움
2. **예측-결과 기록 강제**: reflection 출력에 "예상 / 실제 / 갭" 3필드 필수
3. **세션 종료 강제 consolidation**: `finalize()` 시 자동으로 reflection 프롬프트 진입
4. **콜드 스타트 스캐폴딩**: 첫 세션엔 reflection 장황화 소프트 힌트
5. **토큰 예산**: state 파일은 N 토큰 이하 유지, 압축 책임은 AI에

**원칙**: 시스템은 *트리거*만, AI는 *내용*.

## ADR-003: 액션 모델 = D (full tool-use loop)

**상태**: 채택

**결정**: LLM이 도구(`mouse_move`, `click`, `drag`, `get_screenshot` 등)를 직접 호출하고, `end_turn`을 부를 때까지 반복.

**제약**: 턴당 도구 호출 예산 = 8회 (초기값). 초과 시 시스템이 강제로 `end_turn` 처리 후 다음 사이클로.

**이유**:
- "hover-툴팁-판단-클릭"을 한 턴 내에 처리 가능
- 비용은 LLM이 도구를 활발히 쓸 때만 증가. 적게 쓰면 단일 액션 안과 비슷.

**대안 검토**: A(단일 액션+JSON), B(A+턴내 vision tool), C(액션 시퀀스). D가 hover-재관찰 패턴을 가장 자연스럽게 표현하므로 채택.

## ADR-004: 매 턴 stateless, 모든 기억은 파일

**상태**: 채택

**결정**: LLM 호출 간 롤링 히스토리 없음. 이전 턴 대화는 다음 호출에 안 들어감. AI의 모든 기억은 `meta_notes` / `strategy_note` / `state.{ext}` 파일.

**이유**:
- Voyager / MemGPT 패턴
- 컨텍스트 폭발 방지
- "중요하면 파일에 적어둬라"를 AI에게 명시적으로 강제

---

## ADR-005: 좌표는 픽셀 절대값

**상태**: 채택

**결정**: 정규화 좌표가 아닌 픽셀 `[x, y]` 사용.

**이유**: 고정 창모드이므로 절대좌표가 단순. LLM 좌표 추론 정밀도도 절대값이 안정적.

---

## ADR-006: 모든 액션 후 2초 대기 (Windows 내부)

**상태**: 채택

**결정**: Windows 액션 함수는 *내부에서* 2초 대기 후 반환. 호출자(Mac)는 별도 sleep 불필요.

**이유**:
- 애니메이션·전환·툴팁 렌더링 시간 확보
- 매 액션 일관된 settle time
- 호출자가 매번 sleep 관리할 필요 없음

**경계**: 비전 도구(`get_screenshot`, `get_crop_image`)는 대기 없음.

**제약**: 2초 안에 의미 있게 진행 가능한 게임만 후보로 선정.

---

## ADR-007: 호버 함수 제거, mouse_move + 자동 대기로 합성

**상태**: 채택

**결정**: 별도 `hover(x, y, duration)` API 없음. `mouse_move(x, y)` + ADR-006 2초 대기 = 사실상 호버.

**이유**: 인터페이스 단순화. duration이 고정이라 파라미터 의미 없음.

**고지**: base system prompt에 "마우스 이동 후 2초 내 뜨는 툴팁은 다음 스크린샷에 잡힘"을 명시 (시스템 invariant).

---

## ADR-008: 동기 단방향, 비동기 호환성 미고려

**상태**: 채택

**결정**: Mac↔Windows는 동기 HTTP 요청/응답. 비동기는 PoC 성공 후 별도 검토. 호환성을 위한 추상화는 도입하지 않음.

**이유**: PoC 단순화.

---

## ADR-009: 에러 처리 분리 — transient는 시스템, 의미적은 LLM

**상태**: 채택

**결정**:
- **시스템 레벨 transient** (네트워크 블립, 캡처 실패, 일시적 포커스 손실): Windows/Mac 내부 자동 재시도 (최대 3회). LLM에 안 알림.
- **의미적 에러** (좌표 화면 밖, 클릭 후 화면 무변화 등): 다음 턴 state로 보고. LLM이 판단.
- **3회 retry 한계 초과**: 시스템 에러로 LLM에 보고.

---

## ADR-010: 단일 레포 + 4 폴더 (controller / executor / shared / games)

**상태**: 채택

**결정**: Mac/Windows 코드를 한 레포에 폴더로 분리. 공유 스키마(`shared/schema.py`)를 두 쪽에서 import.

**이유**: PoC 단계에 다중 레포 오버헤드 불필요.

---

## ADR-011: 시스템 invariant는 controller 코드의 base system prompt에 둔다

**상태**: 채택

**결정**: "모든 액션 후 2초 대기" 같은 시스템 계약은 `controller/prompts/` 안 코드 상수로 두고 base system prompt에 합친다. `_shared/meta_notes.md` (AI 갱신 가능 메모리)와는 분리.

**이유**: 시스템 계약과 AI 기억의 권한·출처를 섞지 않기 위함.

---

## ADR-012: 비전 결과는 라벨+좌표 사전, 액션 인자는 픽셀, 결과는 tool_result에 즉시 반환

**상태**: 채택

**맥락**: ADR-001은 "클라우드는 의미 단위(element_id, 라벨)로 액션 결정"이라 했고, spec의 액션 API는 `click(x, y)` 픽셀 인자였다. 둘이 충돌. 또한 D 모델(ADR-003)의 한 턴 내 도구 결과를 LLM이 어떻게 보는지가 명세에 없었다.

**결정**:

1. 액션 도구의 인자는 픽셀 좌표 그대로 유지 (`click(x, y)` 등, ADR-005 그대로).
2. Mac은 매 턴 LLM에게 **라벨 ↔ 좌표 사전**을 user 메시지로 함께 넣는다. 형식 예:
   ```
   element_id  label         box(x1,y1,x2,y2)    click_at(x,y)
   cell_0_0    "빈 칸"        (10,40,50,80)       (30,60)
   tooltip     "마인 3"       (200,300,360,340)   (280,320)
   ```
3. LLM은 사전에서 좌표를 골라 인자에 직접 넣는다. **사전에 없는 자유 좌표 사용도 허용** (빈 공간 클릭, 드래그 destination).
4. 액션 도구 tool_result 표준 페이로드:
   - 성공: `{"ok": true, "executed_at": [x, y], "elapsed_ms": 2031}`
   - 실패: `{"ok": false, "reason": "out_of_bounds" | "failsafe_triggered" | "windows_unreachable" | ...}`
5. **tool_result에 스크린샷 자동 첨부 안 함.** 결과 확인이 필요하면 같은 턴에 LLM이 명시적으로 `get_screenshot()` 호출.
6. "클릭 후 화면 무변화" 같은 *의미적* 성공/실패는 시스템이 판정하지 않는다. ADR-009대로 다음 턴 state에서 LLM이 판단.

**이유**:
- 의미 단위 호출(`click_by_label`)은 사전 신뢰가 100%여야 안전. PoC 단계는 사전을 *참고용*으로 두고 자유 좌표 여지를 남기는 편이 견고.
- 매 액션마다 스크린샷 자동 첨부 = 턴당 최대 8장 추가 → 비용 폭발. on-demand가 ADR-003 정신과 맞음.

**ADR-001 보강**: "의미 단위로 액션 결정" → "라벨+좌표 사전을 보고 좌표 단위로 액션 결정. 사전 참조는 강력 권장이나 강제하지 않음."

## ADR-013: 안전장치 — FAILSAFE + 양측 `/abort` 엔드포인트

**상태**: 채택

**맥락**: LLM이 사용자 물리 머신의 마우스/키보드를 조작한다. 폭주·오인·무한루프 시 사용자가 *반드시* 즉시 정지할 수단이 필요하다.

**결정**: PoC 단계 3중 안전망.

1. **PyAutoGUI FAILSAFE 활성** — Windows executor 부팅 시 `pyautogui.FAILSAFE = True`.
   - 사용자가 마우스를 화면 좌상단(0, 0)으로 빠르게 던지면 즉시 `FailSafeException`.
   - executor는 예외를 잡아 액션 중단 후 Mac에 `{"ok": false, "reason": "failsafe_triggered"}` 응답 (ADR-012 페이로드).
2. **Mac controller `POST /abort` 엔드포인트**.
   - 호출 시 현재 step 중단, 다음 step 차단(`aborted` 상태로 잠금).
   - 재개는 명시적 `POST /resume`. 진행 중 세션은 보존, 사람이 finalize 여부 결정.
3. **Windows executor `POST /abort` 엔드포인트**.
   - 호출 시 abort flag 세팅. 현재 실행 중 단일 액션은 끝까지 진행 후 정지 (PyAutoGUI 호출은 인터럽트 불가 — 단일 액션은 2초 이내 끝남).
   - 이후 모든 액션 요청에 `{"ok": false, "reason": "aborted"}` 응답.

**이유**:
- FAILSAFE는 사용자가 자기 손에 마우스가 있을 때 가장 빠른 수단. OS 핫키 후킹 없이 단순.
- HTTP `/abort`는 LLM 손에 키보드가 잡힌 상황에서 별 머신·핸드폰 curl 한 줄로 정지 가능.

**고지**: base system prompt(ADR-011)에 "FAILSAFE 발동 또는 abort 시 너의 액션은 강제 중단되고 다음 턴 state에 `aborted`로 보고된다"를 명시.

**미정 (후속)**: 글로벌 핫키 인터럽트, 마우스 hijack 방지 데몬 — PoC 이후.

## ADR-014: 컨텍스트 한계 대응 — 단계적 도입 + 명시적 진입 트리거

**상태**: 채택

**맥락**: 게임이 깊어지면 `strategy_note.md` / `meta_notes.md` / 누적 스크린샷이 풍선처럼 커진다. 압축(ADR-002 안전망 5 + 토큰 권장상한)은 *단기* 대응. 깊은 게임의 누적 노하우는 압축만으로 못 잡는다. 큰 컨텍스트 모델로 전부 밀어넣기는 비용·ADR-004 정신 위배.

**결정**: 다음 순서로 점진 도입. 다음 단계는 이전 단계 한계가 *측정 가능한 신호*로 드러날 때만 진입.

**단계 0 (이미 도입됨)**: 토큰 권장상한 + AI 자율 압축 (spec.md "메모리 파일 사이즈 가이드").

**단계 1: 스크린샷 토큰 절감 (직교 기법, 언제든 가능)**
- 풀 캡처 1장만 매 턴 자동 첨부. 추가 영역은 LLM이 ADR-012 라벨 사전을 보고 `get_crop_image`로 명시 요청.
- base prompt에 이 정책 한 줄 명시.
- 도입 비용 낮음 → 단계 2/3 가기 전 우선 시도.

**단계 2: RAG로 strategy_note 분할** (Voyager의 skill library 패턴)
- `strategy_note.md` 단일 파일 → `strategy_notes/` 폴더 + 청크 + 임베딩 인덱스.
- 매 턴 현재 state·스크린샷에 관련된 top-k 청크만 user 메시지에 주입.
- 새 도구 `recall(query)` — LLM 명시적 검색.
- **진입 트리거** (둘 중 하나):
  - `strategy_note.md`가 압축에도 4000토큰 상한을 3 세션 연속 초과.
  - reflection 출력의 "갭" 필드에 "기억 손실로 동일 실수 반복" 류 패턴이 5회 이상 누적.

**단계 3: 세션 간 메모리 인덱싱** (MemGPT 스타일)
- 과거 세션의 `log.md` / reflection 결과 / 스크린샷을 세션 간 임베딩 인덱스로 적재.
- reflection 턴 전용 도구 `recall_past_session(query)` 추가.
- **진입 트리거** (둘 중 하나):
  - 같은 게임 누적 세션 30개 이상.
  - reflection에서 "과거 비슷한 상황을 다시 분석해야 함" 패턴 빈발 (정성 판단, 사람 트리거 허용).

**이유**:
- 단계 1은 cheap·직교적 → 단계 2/3 진입 전 충분히 해결될 수 있음.
- 단계 2/3은 임베딩 모델·벡터 인덱스 인프라 필요. PoC 단계 오버엔지니어링.
- 단계 전환을 측정값에 묶어 *체감*이 아닌 *데이터*로 결정.

**기각된 대안**:
- 큰 컨텍스트(200K~) 모델로 전부 밀어넣기 — 비용 폭발 + 매 턴 동일 토큰 재전송 낭비. ADR-004("기억은 파일") 정신과 맞지 않음.
- 매 턴 시스템 강제 동적 요약 — 정보 손실 누적 + ADR-002("시스템은 트리거만, AI는 내용") 위배.

**미정 (단계 2/3 진입 시 결정)**:
- 임베딩 모델 (로컬 vs 클라우드)
- 벡터 인덱스 백엔드 (FAISS / Chroma / sqlite-vss 등)
- 청크 크기·top-k 값
- `recall` 도구의 턴당 호출 예산 (현 8회 한도와 별개로 둘지)
